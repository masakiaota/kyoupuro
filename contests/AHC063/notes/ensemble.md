# Ensemble Decision

## 目的
- `v012_simple_beam`
- `v139_refactor_v137`
- `v149_no_logs`

上の 3 実装について、十分に軽い問題領域では複数実行し、その中で最も良かった手順を採用する。

## density3 の定義
- `density = M / N^2`
- `density3 = L` : `density < 0.40`
- `density3 = M` : `0.40 <= density < 0.55`
- `density3 = H` : `0.55 <= density`

## 3 実装比較を行う条件
次のどちらかを満たすときに、3 実装比較を行う。

1. `N in {8, 9}`
2. `density3 == L` かつ `N <= 12`

## 実行順
3 実装比較を行う場合の実行順は次で固定する。

1. `v012_simple_beam`
2. `v139_refactor_v137`
3. `v149_no_logs`

## 時間ガード
- `v012_simple_beam` と `v139_refactor_v137` の実行終了時点で、累積実行時間が `0.8s` を超えていた場合は `v149_no_logs` を試さない。
- この場合は、`v012_simple_beam` と `v139_refactor_v137` の 2 本だけで比較して採用を決める。

## 「良かったもの」の定義
- 「良かったもの」とは、3 種の実験の中で絶対スコアを最小化した手順を指す。
- 3 実装比較の対象になった実装のうち、絶対スコアが最小のものを最終出力として採用する。
- `v149_no_logs` を時間ガードで省略した場合は、実際に実行した実装の中だけで比較する。

## 補足
- この ensemble は、厳密な `0.8s` 完全保証よりも、軽い領域での改善を優先した運用である。
- 観測上、上の条件は十分に軽く、`v149_no_logs` の上振れも許容可能と判断した。

## 暫定単独選択ルール
- 問題条件だけで事故領域を細かく切る案も検討したが、現時点では複雑すぎるため採用しない。
- 暫定的に、`N >= 12` の領域では `v149_no_logs` を使う方針とする。
- したがって、ensemble の詳細条件とは別に、単独選択が必要な場面では `N >= 12` を `v149_no_logs` 側へ倒す。

## `N in {10, 11}` の追加実行ルール
- `N in {10, 11}` では、基本方針としてまず `v139_refactor_v137` を実行する。
- `v139_refactor_v137` が `800ms` 以内で終了した場合に限り、続けて `v149_no_logs` を実行して比較する。
- `v149_no_logs` の実行中に、プログラム開始からの累積実行時間が `1.88s` に到達した時点で `v149_no_logs` がまだ終了していなければ、`v149_no_logs` を停止する。
- `v149_no_logs` を停止した場合は `v139_refactor_v137` の解を採用する。
- `v149_no_logs` が最後まで終了した場合は、`v139_refactor_v137` と `v149_no_logs` のうち絶対スコアが小さい方を採用する。
