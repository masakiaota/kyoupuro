# Important Properties

このファイルは、問題の重要な性質を整理する場所である。記号の定義・代表名・型・制約は `notes/notations.md` を参照し、ここでは性質そのものだけを書く。

## 1. A 問題の目標画像には、回転・平行移動で一致する部品が残りやすい

### 主張
A 問題では各反復で `copy` が選ばれる確率が `1 - p / 10` であり、`p ∈ {2, 3, 4, 5}` なので `0.5` 以上 `0.8` 以下である。したがって、目標画像 `g` には、ある部品 `P` とその回転・平行移動版が複数箇所に残っている可能性が高い。

### なぜ成り立つか
`copy` は透明でない画素だけを色付きのまま複製する操作であり、色パターンを崩さずに既存図形を増殖させる。生成の大半が `copy` なら、後段の少数の `paint` や重なりで局所的に変形されても、未改変の共通部分は盤面中に残りやすい。

### 実装上どう効くか
目標画像を 1 画素ずつ再現するより、「再利用できる部品 `P` を掘り出して、作業レイヤーから何度も `copy` する」方針を主軸に置ける。

## 2. 変換 `τ` ごとの一致集合を見ると、後から編集されても残った共通核を拾える

### 主張
変換 `τ` に対して

$$
M_\tau = \{(i, j) \mid g_{i,j} \neq 0,\ \tau(i, j) \text{ が盤面内},\ g_{i,j} = g_{\tau(i,j)} \}
$$

を取ると、`copy` で複製された部品のうち後段でも色が保たれた部分は `M_τ` に含まれる。

### なぜ成り立つか
ある部品 `P` が `τ` によって複製され、その複製元と複製先の両方で最終色が変わっていなければ、対応する各画素は最終盤面でも同色のままである。後から別の `copy` や `paint` が入っても、変えられなかった部分だけは `M_τ` に残る。

### 実装上どう効くか
全変換 `τ` を走査して `M_τ` の 4 近傍連結成分を取り出せば、「剛体変換で複製されたらしい部品候補」をノイズ少なめに抽出できる。局所改変が混ざっていても、完全一致を要求することで安全側に倒せる。

## 3. 目標と一致する部品だけを `copy` する限り、配置順はほぼ自由である

### 主張
各 `occ` が目標画像 `g` と完全一致する部品 `P` の出現であるなら、対応する `copy` を layer 0 にどの順で重ねても、その `copy` 自体が誤色を作ることはない。

### なぜ成り立つか
`copy` が上書きするのは source の非透明画素だけであり、ここに置かれる色がすべて `g` と一致しているなら、未塗りセルを正しく埋めるか、すでに正しいセルを同じ色で再上書きするだけである。

### 実装上どう効くか
候補部品の選択は「残差 `R` をどれだけ減らせるか」に集中できる。複数部品の順序依存を細かく考え込まず、まずは貪欲または局所探索で採用集合を決め、その後に残った `R` を `paint` で埋めればよい。

## 4. `N = 32` 固定なので、全変換の総当たりは十分軽い

### 主張
候補抽出で必要な「回転 4 通り × シフト `(2N - 1)^2` 通り」の全探索は、`N = 32` では現実的である。

### なぜ成り立つか
変換総数は

$$
4 \times (2N - 1)^2 = 4 \times 63^2 = 15876
$$

しかない。各変換で高々 `N^2 = 1024` 画素を調べても、単純な一致判定は約 `1.63 \times 10^7` セル比較で済む。

### 実装上どう効くか
候補部品の抽出に近似やサンプリングは不要である。全変換を一度なめて部品候補を作り、その後は上位候補だけに対して詳細な出現列挙やプラン評価を行えばよい。

## 5. レイヤー数 `K` は小さいので、浅い再利用ライブラリとして設計するのが自然である

### 主張
`K <= 5` なので、多数の作業レイヤーを同時に保持する大規模な文法展開より、「少数の部品を順に準備して layer 0 に吹き付ける」設計の方が自然である。

### なぜ成り立つか
最終出力に必須なのは layer 0 だけであり、残りは最大 4 枚しかない。深い再帰展開や多数の部品の常駐は、`clear` コストと管理複雑度に見合いにくい。

### 実装上どう効くか
初版は「1 枚の部品レイヤーを作って何度も `copy` し、必要なら `clear` して次の部品に切り替える」だけでよい。`K >= 3` のときだけ 1 枚補助レイヤーを使って `build_cost(P)` を再帰的に圧縮する拡張を入れれば十分である。

## 6. exact prep copy は work layer 上で誤色を作らない

### 主張
source pattern の exact occurrence だけを使って target pattern を構築する限り、work layer 上の `copy` は target 外の誤色を作らず、残差は不足 `paint` だけで埋めればよい。

### なぜ成り立つか
prep recipe では、source pattern の各 occurrence が target pattern 内の同色セル集合と完全一致するものだけを採用する。したがって `copy` で置かれる非透明セルはすべて target の正しいセルである。

### 実装上どう効くか
work layer 上の構築では approximate match を考えず、`copy` の利得判定を「何セルを exact に置けるか」に単純化できる。repair は layer 0 吹き付けだけに残せばよい。

## 7. depth 2 の再利用は beam 上の `Load` 連鎖で十分表現できる

### 主張
`R -> Q -> P` のような浅い再利用は、再帰 recipe を持たなくても beam 上で `Load(Q)`, `Load(P from Q)`, `Spray(P)` を順に積めば表現できる。

### なぜ成り立つか
各 work layer は「空」か「完成済み pattern」だけを状態に持てばよい。source layer を残したまま別 layer に target を組めるので、浅い依存 DAG は action 列で自然に表現できる。

### 実装上どう効くか
recipe 自体は 1 段 exact compose に限定し、再帰展開は beam の状態遷移に任せれば、状態表現と replay 実装を単純に保てる。

## 8. 残差 `R` を最後に `paint` すれば、常に baseline 以下には落ちない

### 主張
どの段階でも、採用した部品 `copy` の後に残った `R` をすべて `paint` すれば正解は作れる。したがって、見積もりコストが悪い部品は捨てて baseline に戻せる。

### なぜ成り立つか
`paint(0, i, j, g_{i,j})` を非透明画素すべてに対して行えば、操作回数 `U` で必ず正解できる。`copy` を使う方針はこの baseline を下回るときだけ採用すればよい。

### 実装上どう効くか
プラン評価は「推定総手数が `U` 未満か」を最低条件にしてよい。これにより、攻めた候補探索をしても最終的な安全策は失われない。

## 9. pattern の良し悪しは時系列の局所判断ではなく、順序付き適用列として事前最適化できる

### 主張
`copy` 中心の solver では、最終操作列は時系列だが、本質的には「どの pattern をどの variant でどの順に使うか」という高水準の列 `S` を先に最適化できる。

### なぜ成り立つか
各 step の作用は現在の `R` に対する batch 適用として定義でき、最終的な手数 `T` と score は `S` を前から replay するだけで評価できる。したがって、操作列を直接いじらずとも、`S` を焼きなましで探索できる。

### 実装上どう効くか
近傍は `Insert / Delete / Move / ReplacePattern / MutateVariant / Duplicate` のように高水準に設計できる。評価関数には問題本来の score をそのまま使ってよい。
