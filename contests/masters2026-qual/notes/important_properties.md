# Important Properties (masters2026-qual)

このファイルは、解法設計で効く「重要な性質」を蓄積するためのメモである。

## 0. 記号
- `K`: ロボット台数
- `M`: 全ロボットの内部状態数合計
- `W`: 追加壁本数
- `V = AK * (K - 1) + AM * M + AW * W`

## 1. 差分コストの基本式
- ロボット1台を新規追加し、そのロボットの状態数が `m_new` のとき:
  - `ΔV_add_robot = AK + AM * m_new`
  - 制約 `m_new >= 1` より、`ΔV_add_robot >= AK + AM`。
  - 特に A (`AK=0, AM=1`) では、ロボット1台追加の下限コストは `+1`（完全無料ではない）。
- 既存ロボットに状態を `Δm` 追加するとき:
  - `ΔV_add_state = AM * Δm`
- 壁を `ΔW` 本追加するとき:
  - `ΔV_add_wall = AW * ΔW`

## 2. 「新規ロボット vs 状態追加」の比較
- 新規ロボットが得:
  - `AK + AM * m_new < AM * Δm`
- 既存への状態追加が得:
  - `AM * Δm < AK + AM * m_new`
- 注意:
  - 判定は `AK` と `AM * m_new` だけでは不十分で、既存側に必要な `Δm` が本質である。

## 3. 周期尾部のみ有効 + 機能グラフ構造
- 採点で警備対象に数えられるのは、各ロボットの周期部分 (`tail`) に含まれるマスのみである。
- 過渡部分 (`head`) だけで訪れるマスは無効である。
- 各ロボットの状態は `(pos, dir, internal_state)` で表せ、遷移は決定的なので出次数1の機能グラフになる。
- 初期状態から到達する有向サイクルは1つだけであり、警備対象は「その1サイクル上のノードの `pos` 射影」に一致する。
- したがって実体は「各ロボット1サイクルのセル集合の被覆問題」である。

## 4. 問題別の大局方針
### A (AK=0, AM=1, AW=1000)
- 実質 `V = M + 1000W` なので `W=0` が基本。
- 目標は `M` 最小化。
- `AK` 項による `K` 罰則はないが、`m_k >= 1` のため台数を増やすと最低でも `M` が増える。
- よって追加ロボットは無料ではなく、「台数増で他ロボットの状態数をそれ以上に削れるか」で判断する。

### B (AK=1000, AM in [1,10], AW in [1,10])
- 入力ごとに `AM/AW` が変わるので、方針を動的切替する。
- ロボット実行中に乱数は使えない（制御は決定的）。
- 壁は安いことが多く、`W` を増やして `M` や `K` を下げる戦略が有効になりやすい。
- `AM/AW` を見て「状態削減のための壁追加」を積極化する。
- 候補操作の受理条件は常に `ΔV < 0` とし、
  - `ΔV = AK * ΔK + AM * ΔM + AW * ΔW`
  - を満たす操作だけ採用する。

### C (AK=1000, AM=1, AW=1000)
- `K` も `W` も重いので、まず `K=1, W=0` を軸に探索する。
- 追加ロボット/追加壁は、`M` を大きく削れるときのみ検討する。

## 5. 実装時の意思決定ルール（メモ）
- 共通受理条件:
  - `ΔV = AK * ΔK + AM * ΔM + AW * ΔW`
  - `ΔV < 0` のときのみ候補操作を採用する。
- 追加ロボット採用条件:
  - `AM * (M_before - M_after) > AK`
- 壁追加採用条件:
  - `AM * (M_before - M_after) + AK * (K_before - K_after) > AW * ΔW`
- 壁追加は非単調:
  - 壁を増やすと既存の良い周期が壊れ、`M` や `K` の再最適値が悪化する場合がある。
  - したがって壁の評価は「壁だけの局所効果」ではなく、再最適化後の `(ΔK, ΔM, ΔW)` 全体で判定する。
- 実装上は、上式を「局所操作の受理判定」として使う。

## 6. 仮説
- サイクル領域の重複が大きい解は、同じ `ΔV` あたりの未被覆セル純増が小さくなりやすく、総じて不利になりやすい。
  - 記号 `U` を既存被覆セル集合、候補サイクルセル集合を `C` とすると、実効寄与は `|C \\ U|` で決まる。
  - `|C ∩ U|` が大きい候補は、同コストなら `|C \\ U|` が小さくなり、採用優先度が下がるはず。
  - ただし狭隘通路や接続維持の都合で重複が不可避なケースはあり、常に悪いとは限らない。
- B/C では「1台で長い蛇行サイクルを作る」初期解が強い可能性が高い。
  - `AK` が重く増台コストが高いため、まず `K=1` で `M` を詰めるのが有利になりやすい。
  - A では逆に多台数小状態の方が有利になりやすい（別方針）。
- 段階戦略仮説:
  - 1) まず `K=1` で最大被覆 + 低 `M` を探索する。
  - 2) 未被覆セルが残り、1台での改善余地が乏しいときだけ `K` を1台ずつ増やす。
  - 3) `K` を固定した後にオートマトン最小化（状態マージ/冗長遷移削減）を行う。
  - 4) 最終段で WA 回避のための救済増台を許容する。
- 追加ロボット採否の実用判定:
  - `AK + AM * m_new < AM * Δm_saved` のとき採用。
  - 特に `m_new = 1` なら `Δm_saved > AK / AM + 1` が目安。
- 壁追加（主に B）での改善仮説:
  - 壁で遷移を単純化し `M` を下げられるなら有効。
  - 受理条件は再最適化込みで `ΔV = AK * ΔK + AM * ΔM + AW * ΔW < 0`。
  - `K` 増を誘発する壁は `AK` が重い問題で不利になりやすい。

## 8. A/B/C で実際に効いた特性（v003〜v006, stride4_25）
`results/summary.csv` の `stride4_25` 比較より、以下が観測できる。

- A 平均スコア:
  - v003 `4,031,656` -> v005 `4,272,711` -> v006 `4,342,926`（改善） (ちなみにこれもatcoder上のjudgeだとv006は悪化だった)
- B 平均スコア:
  - v003 `7,089,882` -> v005 `7,490,778`（改善）-> v006 `6,725,832`（悪化）
- C 平均スコア:
  - v003 `9,211,708` -> v005 `9,610,279`（改善）-> v006 `8,454,451`（悪化）

### A で効く特性
- 本質は `M` 最小化である（`AK=0, AM=1, AW=1000`）。
- 1台長周期より、多数の小さな周期候補を集めて集合被覆で組む方が強い。
- 候補多様化（`m=1,2` 全列挙 + `m=3,4` サンプリング）と、選択後の局所改善（追加/削除）が効く。
- `K` 自体に罰則はないが、`m>=1` なので「台数を増やせば自然に有利」ではない。`M` 減少が伴う増台だけが有効である。

### B で効く特性
- `AK=1000` が重く、増台が効きにくい。まず `K=1` を軸にするのが安定する。
- 改善の主戦場は「1台ルートの圧縮」であり、`2-opt`/`or-opt`/キックで巡回長 `M` を削る戦略が有効である（v004/v005 で改善）。
- A向けの「増台トレードオフ」をそのまま持ち込むと、`K` 増加のペナルティで失速しやすい（v006 で顕在化）。
- `AW` が軽いケースでは壁も候補になるが、必ず再最適化後の `ΔV = AKΔK + AMΔM + AWΔW` で判定すべきである。

### C で効く特性
- `AK=1000, AM=1, AW=1000` なので、`K` と `W` を増やす余地がほぼない。
- 実戦では `K=1, W=0` 固定で、単一ルートの `M` を詰める戦略が最も安定する（v003->v005 で改善）。
- Cは `AM=1` のため、少しの構造悪化でも `M` 増がそのまま失点に直結する。増台なしでもルート品質悪化で簡単に崩れる（v006）。

## 9. 実装指針（現時点）
- A:
  - 候補集合を厚くして集合被覆 + 局所改善を回す。
  - 制限時間は主に「候補生成数」と「被覆改善反復」に投資する。
- B:
  - 基本は `K=1` ルート探索の近傍強化に時間を使う。
  - 壁導入は別フェーズに分け、`ΔV<0` を厳密に満たすときのみ採用する。
- C:
  - `K=1, W=0` を崩さず、ルート長最小化だけに計算資源を集中する。
  - A型の増台・トレードオフ探索は原則オフにする。

## 10. v101〜v103 で確定した知見（壁 + 良質オートマトン）
- 壁は「曲がりトリガ」として使うと効く。
  - 直進区間を自己ループ化し、方向転換が必要な箇所だけ壁で遷移を発火させると、`M` を大きく削れる。
  - 特に B では `AW` が軽いケースが多く、`AW * ΔW < AM * ΔM (+ AK * ΔK削減分)` を満たしやすい。
- 壁は入れれば良いわけではない。
  - 追加可能壁の一括採用は過剰壁になりやすく、`ΔV` 悪化を招く。
  - 壁ごとに「圧縮で減る状態数の合計」を見積もり、`ΔV<0` の壁だけ採用するのが有効。
- C では壁は重く、同じ戦略でも最終的に `W=0` が選ばれやすい。
  - つまり「壁戦略の実装」は C でも必要だが、「採用される壁数」は入力係数で自然に抑えられる。
- 良質オートマトンのテンプレ群を併用すると、B/Cだけでなく A も改善し得る。
  - `route/wall` 系ベースラインとテンプレ被覆解を競合させ、`V` 最小を選ぶハイブリッドは安定して強い。
  - B の伸びが C より小さいのは、v102時点で B が既に壁圧縮でかなり詰まっており、追加改善余地が相対的に小さいためと解釈できる。

## 11. 次の検証優先度（提案）
- P0: オートマトン探索を30分級に拡張（最優先）
  - 狙い: テンプレ品質の底上げ。
  - 施策: `max_m` 拡大、`trials_per_m` 増加、複数seedで多様化、保存上限を拡張。
  - 判定: 25ケースで `v103` 比 `+1%` 未満なら探索設定を見直す。
- P0: A専用テンプレ探索を実施
  - 狙い: Aの集合被覆候補に「小状態で被覆効率の良い周期」を追加。
  - 施策: A評価関数（`W=0` 前提）でテンプレ採点し、B/C用とは別ライブラリを作る。
  - 判定: Aの `M` 合計改善が出るかを最優先で見る。
- P0: B/C ライブラリの相互乗り入れ検証
  - 狙い: B由来テンプレがCで強い、またはその逆を拾う。
  - 施策: `library_B only`, `library_C only`, `B+C union` の3条件で比較。
  - 判定: 問題別ベストの採用率と合計スコア差分を記録。
- P1: Bの壁候補生成を「曲がりたい地点」中心に再設計
  - 狙い: 少ない壁で大きい `ΔM` を取る。
  - 施策: ルート上の高頻度ターン点、長直線端点、ボトルネック近傍だけを壁候補にし、局所探索で採否決定。
  - 判定: `W` 減少と `M` 減少の両立（`V` 純改善）を確認。
- P1: Bの乱数係数 (`AM, AW`) に対するロバスト化
  - 狙い: 係数ブレに強い方策選択。
  - 施策: 入力ごとに `(AM,AW)` からモード切替（壁積極/壁抑制/テンプレ優先）し、境界条件を学習。
  - 判定: 係数帯ごとの勝率ヒートマップで確認。
- P2: 過学習抑制のための検証分割
  - 狙い: `stride4_25` 固定最適化の偏りを検出。
  - 施策: 25ケースを固定したままでも、探索用/比較用の分割評価を回し、改善が再現するかを見る。
  - 判定: holdout側で改善が消える案は不採用にする。

## 12. 追加で考えられる改善点（今回の議論に上乗せ）
- テンプレの「連結合成」探索:
  - 複数の性質の良い小オートマトンを、少数のゲート状態で接続する方式を探索対象にする。
  - 1台のままモード切替できるため、`K` を増やさずに被覆多様性を得られる可能性がある。
- 近傍探索の目的関数を多目的化:
  - 直接 `V` だけでなく、`未被覆数` と `将来の圧縮余地`（推定 `ΔM`）を併置して候補を保持する。
  - 局所最適への早期収束を抑えられる可能性がある。
- テンプレ採用の説明変数ログを残す:
  - どの入力特徴（障害物密度、連結成分形状、直線通路長分布）でどのテンプレが勝つかを記録する。
  - 次段で「入力特徴 -> 初期候補選択」の学習的ルール化ができる。

## 13. v104で追加で確かめた性質
- 問題別に「最適な相互乗り入れ比率」は異なる。
  - Bは `B由来ライブラリ` の比率を高くすると伸びやすく、Cは `C由来ライブラリ` の比率を高くすると伸びやすい。
  - ただし全問題で同一比率を使うと、どれかが悪化しやすい。
- したがって、A/B/Cを1本の設定で縛るより「問題別ベスト構成を採用」する方が強い。
  - 実際に `v104_mix1`（A=try1, B=try1, C=try2）が単一設定版より高得点だった。
- 探索時間延長（B/C `SEARCH_TIME_MS` 増加）は、テンプレ解が優勢なとき効果が薄い。
  - route探索時間を増やしても最終採用解が変わらないケースではスコア差が出にくい。

## 14. v106で確かめた性質（斜め優先）
- 周期内で「2手先が斜め隣」になる頻度（`diag2`）を優先すると、A/B/Cすべてで改善が出た。
  - 直感的には、蛇行で面積効率よく被覆できるオートマトンが上位に残りやすくなる。
- ただし斜め優先だけに寄せると悪化しうる。
  - 従来強いテンプレ群と混合したときに最も安定して伸びた。
- 斜め性は「追加の評価軸」として効く。
  - 既存の被覆・周期長最適化を置き換えるのではなく、補助目的関数として使うのが有効。
